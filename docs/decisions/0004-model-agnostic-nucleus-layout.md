# ADR 0004: Model-Agnostic Nucleus Layout (nucleus / adapters / distributions)

## Status

Accepted.

## Context

Today canonical source, harness adapter templates, generated plugin output, and
docs are mixed across `omp/.omp/agent/`, `.agents/skills/`,
`docs/harness/codex-adapter-plan/templates/`,
`docs/harness/claude-adapter-plan/templates/`,
the plugin-bridge tree under `docs/harness/`, the OMP built-ins snapshot under `docs/harness/`,
`scripts/`, and `tests/`.

`render-harness-nucleus.mjs` is a ~979-line monolith holding candidate model,
parsers, gate, apply engine, marker I/O, and reporting.
`render-plugin-bridge.mjs` imports `applyCandidates`/`saveMarkerIfChanged`/
`resolveDisposition` from it, and the shared
`~/.loom-harness/applied-manifest.json` marker records
`generatedBy: render-harness-nucleus` even for plugin-bridge writes.

Path literals are copy-pasted instead of imported: the
`docs/harness/resource-manifest.json` default appears in 5 scripts
(`dry-run-harness-inventory.mjs:7`, `dry-run-harness-safety-gate.mjs:33`,
`validate-harness-manifest.mjs:6`, `render-harness-nucleus.mjs:51`,
`render-plugin-bridge.mjs:42`); `.agents/skills` in 5 scripts;
Codex/Claude plan paths in 3 scripts each; the omp source root in 4+.

`.agents/skills/` is Loom's own live repo-local skill root: the OMP harness
discovers repo-local skills there, and the pre-ADR plugin-bridge tree
carried generated byte-copies of it checked by
`validate-shared-agent-packages.mjs`.

`python3` >= 3.11 with `tomllib` is an undocumented hard dependency of the
render gate (`render-harness-nucleus.mjs:484-499` shells
`python3 -c "import tomllib"`); a follow-up issue (LOO-114) owns the parser
strategy decision.

ADR 0002 decided Loom is a skill/harness library with no downstream
`docs/agents/` project layer. ADR 0003 decided clean cutover / no second
convention, and that the kit is Linear-first with the tracker binding per repo.

## Decision

1. **Lanes.** Allowed top-level lanes are `nucleus/`, `adapters/`,
   `distributions/`, `docs/`, `scripts/`, `tests/`, plus `.agents/` retained
   only as a rendered compatibility surface. No other top-level source lane is
   allowed.
   - `nucleus/` is the model-agnostic canonical source for agents, skills,
     rules, workflows, and schemas.
   - `adapters/{omp,codex,claude,generic}/` contains harness-specific source,
     templates, and renderer entrypoints.
   - `distributions/` contains generated/checkable output, including
     `loom-nucleus` plugin output and reference snapshots.
   - `docs/` contains explanations, architecture decisions, manifests, and
     reference docs only; no templates and no generated payloads.

2. **Model-agnostic defined.** For `nucleus/`, model-agnostic means no
   harness/provider/model coupling: no OMP, Codex, or Claude references; no
   model names; no provider routing; no live home paths (`~/.omp`, `~/.codex`,
   `~/.claude`); and no adapter install paths. Tracker references
   (Linear/GitHub) remain allowed where ADR 0003's Linear-first kit requires
   them. This ADR is harness-agnostic, not tracker-agnostic; it does not
   supersede ADR 0003's tracker stance.

3. **`.agents/` resolution.** `.agents/skills/` stays as a rendered
   compatibility surface generated from `nucleus/skills/`, because it is the
   live repo-local skill root the OMP harness reads. It is not a second
   canonical source; edits land in `nucleus/` and render outward.

4. **Central layout module.** A single module, `scripts/lib/layout.mjs`,
   exports every lane root and derived default path: resource manifest, skills
   root, Codex/Claude plan paths, and omp source root. Renderers, validators,
   and tests must import these instead of copy-pasting literals. This module
   lands before any renderer/test rewiring (LOO-111 step 0).

5. **Renderer seams.** `render-harness-nucleus.mjs` decomposes along four named
   seams: candidate model; render/gate; apply engine; reporting. LOO-111
   implements the split; this ADR fixes the seam names.

6. **Marker ownership.** Each renderer entrypoint stamps its own identity in the
   shared applied-manifest marker: `generatedBy` must name the actual writer,
   such as `render-plugin-bridge`, never inherited from a shared library
   default. Marker file format stays shared; identity is per-writer.

7. **python3/TOML stance.** This ADR records the dependency surface but does not
   fix the parser choice. LOO-114 decides whether to keep the python3 check,
   vendor a parser, or document the dependency. Whatever LOO-114 decides must
   be reflected in doctor/setup docs; nothing in this ADR mandates python3
   permanently.

8. **Doc supersession.** Path language in `docs/architecture/harness-bridge.md`,
   `docs/harness/resource-manifest.md`, `docs/harness/codex-adapter-plan.md`,
   `docs/harness/claude-adapter-plan.md`, and
   `docs/harness/plugin-bridge/design.md` is superseded by this ADR's lane map
   where the two conflict. The doc rewrites themselves land in the cleanup issue
   (LOO-112), not here.

9. **ADR 0002 and 0003 preserved.** Loom stays a skill/harness library with no
   `docs/agents/` layer (0002 intact). Clean cutover / no-second-convention
   (0003) governs the migration: when a file moves lanes, the old path dies in
   the same PR series. No aliases, no duplicated trees left behind, and
   byte-equality validators bridge only during the move window.

## Rejected Alternative

- **Tracker-agnostic nucleus** (scrubbing Linear/GitHub mentions from
  `nucleus/`): rejected; it contradicts ADR 0003's Linear-first kit and buys
  nothing. Trackers are a planning binding, not a harness coupling.
- **Deleting `.agents/skills/` in favor of `nucleus/skills/` only**: rejected;
  OMP harness discovery reads `.agents/skills/` live, so the compatibility
  surface must stay rendered.
- **Keeping per-script path literals with a lint rule instead of a layout
  module**: rejected; a lint cannot feed tests/renderers a single source of
  truth, and duplicated defaults have already carried drift risk too long.
- **Splitting the renderer opportunistically without named seams**: rejected;
  unnamed seams re-create a monolith under new filenames.

## Consequences

LOO-107..110 (moves) and LOO-111 (rewiring) implement this layout. LOO-112 cleans
docs. LOO-113 consolidates duplicated safety-policy logic into `scripts/lib/`.
LOO-114 decides the TOML parser.

Until the moves land, the mixed layout persists; new files must already follow
the lane map.

If Loom ever needs harness-specific content inside `nucleus/`, this ADR must be
superseded.
